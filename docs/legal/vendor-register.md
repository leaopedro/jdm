# Vendor Register

Canonical vendor-onboarding and annual-review register for JDM Experience.
This file satisfies the hard onboarding gate in
`docs/legal/vendor-management-policy.md`.

Usage rules:

- Add or update a row before introducing any new vendor, SDK, hosted runtime,
  webhook integration, or service account.
- Keep unresolved legal, contract, or security facts explicitly marked as
  `Pending`.
- Re-review every approved vendor at least once every 12 months or whenever the
  integration scope expands.

Scope note:

- This register captures the 12 vendors already enumerated in `LGPD_scan.md`
  section 7.
- Several vendors predate the existence of this formal register. Their
  `Onboarding date` below records when the register row was first formalized for
  governance, not necessarily the original production-adoption date.

## Art. 33 transfer mechanism (T09 / Q06)

The operative rule for international transfers under LGPD Art. 33 is recorded
in the [JDMA-631 Q06 decision comment](/JDMA/issues/JDMA-631#comment-f0f8ae6c-8f54-46c9-b610-300b98135793).
It governs every row above whose `Country/region` is outside Brazil.

Priority order per vendor:

1. **ANPD Res. 19/2024 Anexo II SCC text, incorporated verbatim** as a DPA
   addendum where the vendor signs custom riders.
2. **Vendor's own DPA only if it incorporates Res. 19/2024 SCC text by
   reference** or contains equivalent clauses verified clause-by-clause
   against Anexo II.
3. **Art. 33, IX — explicit, specific, separated titular consent** for
   vendors that will not sign any SCC instrument (OAuth providers).
4. **No mechanism required** for vendors processing data only inside Brazil
   under Art. 33.

No transfer may proceed under Art. 33, II (international cooperation) or any
other hypothesis without counsel sign-off recorded on the vendor's row.

### Per-vendor disposition

Posture legend (used consistently in the table and the outreach SLA below):

- `GO` — Q06 mechanism is final and the vendor already publishes a compatible
  DPA, addendum, rider, or portal path. Execution work is reduced to retrieving
  the artifact, attaching the Anexo II schedule where applicable, and filing
  the signed PDF. No 30-day vendor-acceptance gate.
- `CONDITIONAL` — Q06 mechanism is final but vendor acceptance must be
  confirmed via vendor support within 30 days. If the vendor declines or does
  not respond inside that window, the row moves to a `replace` decision per
  its escalation rule.
- `CONSENT` — Art. 33, IX titular consent. The vendor will not sign any SCC
  instrument; the row is GO once the consent-capture UX ships.

`GO` vendors still need their signed DPA snapshot filed at the row's evidence
path before launch. The escalation rule on a GO row only fires if the vendor
unexpectedly retracts a previously published instrument, not as part of the
30-day SLA.

### Outreach SLA and escalation

The 30-day vendor-acceptance SLA applies **only to `CONDITIONAL` rows**:

- `CONDITIONAL` vendors — **Resend, Expo Push, EAS Build, Vercel** — receive a
  single legal outreach within 30 days of T09 closeout. The outreach asks the
  vendor to either sign the ANPD Anexo II addendum verbatim or confirm that
  the vendor's own DPA already incorporates equivalent clauses
  clause-by-clause.
- If a `CONDITIONAL` vendor refuses or does not respond inside the 30-day
  window, the row moves to a `replace` decision and the escalation rule for
  that vendor is opened as a sourcing issue.
- `GO` vendors — **Stripe, AbacatePay, Cloudflare R2, Sentry, Railway, GitHub
  Actions (Microsoft)** — do **not** have a 30-day acceptance SLA because the
  vendor already publishes a compatible DPA / portal path. Their execution
  work is to download the current DPA snapshot, attach the Anexo II schedule
  where applicable, file the signed PDF at the evidence path, and confirm no
  silent vendor-side change has retracted the published posture. The
  per-row escalation rule only fires if that confirmation surfaces a vendor
  retraction.
- `CONSENT` vendors — **Google OAuth, Apple Sign In** — cannot accept any SCC
  instrument; they are not on the 30-day SLA. They become GO under Art. 33,
  IX once the consent UX in [JDMA-649](/JDMA/issues/JDMA-649) and
  [JDMA-668](/JDMA/issues/JDMA-668) ships and the privacy notice names each
  provider, purpose, and destination country.
- Domestic-only execution paths (AbacatePay, Railway data plane) require no
  Art. 33 hypothesis but still need a signed vendor DPA filed at their
  evidence path before the row leaves `Pending` in the main table.

### Evidence storage rules

- Every signed DPA, Anexo II rider, or vendor confirmation is filed at the
  path noted in the disposition table above. Binary artifacts land in
  `docs/legal/vendors/<vendor>-dpa.pdf` via git-lfs once the directory is
  provisioned by the outreach follow-up issue.
- Until the binary lands, the matching `Contract status` cell in the main
  table stays `Pending` so the onboarding gate continues to require
  resolution.
- If a vendor will not accept ANPD Anexo II verbatim and the Q06 priority-2
  clause-by-clause review fails, the row records the explicit `replace`
  decision plus the sourcing issue link.

### Verification status at T09 closeout

At this commit, the register reflects the **final mechanism per vendor** in
the Q06 hierarchy. Posture buckets and the resulting execution work split as
follows; signed PDFs do not yet exist for any row and the `Outreach status`
column states this explicitly per vendor:

- `GO` (mechanism final, no acceptance gate; artifact retrieval pending) —
  Stripe, AbacatePay, Cloudflare R2, Sentry, Railway, GitHub Actions
  (Microsoft).
- `CONDITIONAL` (mechanism final, 30-day vendor-acceptance SLA; replacement
  on refusal) — Resend, Expo Push, EAS Build, Vercel.
- `CONSENT` (Art. 33, IX; depends on consent UX shipping) — Google OAuth,
  Apple Sign In.

Execution of the 30-day SLA, the filing of signed DPAs for every row, and
the consent-UX gating for `CONSENT` rows is tracked as the T09 follow-up
issue.
